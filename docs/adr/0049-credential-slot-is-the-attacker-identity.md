# Credential slot is the rotation-stable attacker identity; secrets and TestCase key on it

The attacker in an authz `TestCase` is identified by **`(principal_label, slot)`** ‚Äî a rotation-stable handle for one of a declared Principal's credentials ‚Äî not by the content-addressed `auth_context_id`. A **credential slot** is a tester-declared label on `DeclaredAuthContext` (`slot: session`, `slot: stepup`, ‚Ä¶) that defaults to the `token_kind` when omitted, is persisted on the `AuthContext` node by the loader, and is copied forward by the auth-helper on every rotation. `(principal_label, slot)` is unique per engagement (the loader rejects duplicates).

Three things key on the slot:

1. **Secrets lookup.** A `SlotResolvingSecretStore` builds, once at run-arm, a graph map `{declared auth_context_id ‚Üí (principal_label, slot)}` over **every** `tier='declared'` AuthContext on the engagement (all generations, `status ‚àà {active, expired}`). `material_for(stale_id)` resolves `id ‚Üí (label, slot)` then returns the freshest material for that slot ‚Äî checking the rotation-file overlay first, then the env-derived store. The dispatcher is correct whether or not the auth-helper is running.
2. **Rotation file + helper `managed`.** The rotation file is `{f"{label}:{slot}": {raw, kind}}`; the helper writes **one** entry per rotation (the old‚Üînew dual-write is removed). The helper seeds `managed: dict[(label, slot), ManagedAuthContext]` from the **graph** at start (not from its env), so its reactive `auth_invalid ‚Üí rotate` path covers every historic id for the slot via the same map.
3. **TestCase `key_hash`** (amends ADR-0007). `auth_context_id` is replaced by `(attacker_principal, attacker_slot)` in the canonicalised hash input. `auth_context_id` stays on the node as a **non-key evidence property**, updated `ON MATCH SET` to the AC the most recent planner run held. Anonymous tests use `("anonymous", "anonymous")`. ADR-0007's intent ‚Äî "same test under a different auth state is a distinct TestCase" ‚Äî is preserved: a different *slot* is a different auth state (C4 weak/strong), a rotated token in the *same* slot is not.

## Why

`auth_context_id = sha256(engagement_id, sha256(kind, canonical(token)))` is content-addressed on the token bytes (ADR-0015). Session cookies rotate. On a real cookie-auth staging engagement (three declared principals, `kind=cookie`), every approved TestCase's `auth_context_id` was unreachable by dispatch time: the loader hashed the engagement-start cookie, the planner committed against that id, `login-*.sh` regenerated the env file, and `EnvSecretStore` (which re-hashes the env at run-arm) indexed a different id. 13/13 TestCases drained as `hazard_unresolved: no live token material`. The auth-helper's old‚Üînew bridge only spans rotations *it* performs; a manual refresh before helper-start is invisible to it because `managed` was env-keyed.

The content-addressed `AuthContext` is correct for the **graph** ‚Äî provenance, `HIT` edges, and identity reconciliation hang off the exact credential observed. But the **attacker** in an authz test is the actor-at-a-capability ("`admin` with the session cookie"), not a token snapshot. Slot is that actor-at-a-capability handle: tester-declared (an ADR-0012 tester-side fact, not black-box seeding), stable across rotation, and fine-grained enough to keep C4's weak/strong distinction (different slots) while collapsing rotation churn (same slot).

## Considered Options

- **Principal-only indirection** (`(principal_label, kind)`) ‚Äî rejected: collides when a Principal declares two credentials of the same `kind` (primary session cookie + step-up cookie; two API keys at different scopes). Wrong credential spliced into a C4 test = false result.
- **Capability tier as the discriminator** ‚Äî rejected: per ADR-0039 capability tier is *inferred* from claim deltas, not a stored property; making it tester-declared is the kind of declarative seeding the black-box discipline avoids. Slot is the tester-side fact ("I have two credentials"); capability ordering between slots stays an inference.
- **Auth-helper as the only bridge** (helper bootstraps from graph, immediately rotates every slot, writes under all historic ids; dispatcher stays id-keyed) ‚Äî rejected as the *primary* mechanism: makes dispatch correctness depend on the helper being up, and forces an IdP round-trip per slot at helper-start even when env tokens are live. ADR-0014's trust split gives the helper *refresh* capability, not *resolution* capability. Kept as a secondary: helper seeds `managed` from the graph so its reactive path is total.
- **Keep `auth_context_id` in `key_hash`; accept re-propose churn** ‚Äî rejected: every planner re-run after a rotation mints a duplicate TestCase for the same logical test, defeating ADR-0007's dedup and ADR-0040's `disposition: permanent` rejection (different key). With slot in hand the logical identity is rotation-stable; the content-address should match it.
- **Normalise `auth_context_id` to the slot's current `status=active` AC at commit time** ‚Äî rejected: races with mid-commit rotation, and the stored id is no longer the AC the planner actually had evidence for.
- **Operational discipline only** (re-run `engagement start` after every cred refresh) ‚Äî rejected: session cookies on a real target rotate faster than a human review cycle.

## Consequences

- **`setup/config.py`**: `DeclaredAuthContext.slot: str | None`, validator defaults it to `kind`; `EngagementConfig` model validator rejects duplicate `(principal.label, slot)` per engagement.
- **`setup/loader.py`**: persists `slot` on the declared `AuthContext` node.
- **`dispatch/secrets.py`**: `EnvSecretStore` gains a `by_slot: dict[tuple[str, str], AuthMaterial]` index; rotation-file format becomes slot-keyed; `SlotResolvingSecretStore(graph_map, rotation_path, env_store)` subsumes `RotatableSecretStore`. `material_for(anonymous_id)` short-circuits to a no-auth sentinel.
- **`dispatch/cli.py`**: one Cypher at run-arm builds `graph_map`; constructs `SlotResolvingSecretStore`. The `hazard_unresolved` reason string drops the misleading "(declared principals only; check `${VAR}`)" hint.
- **`dispatch/auth_helper.py`**: `from_config` seeds `managed` from a graph query over declared ACs (grouped by slot), not from env; `_rotate_graph` sets `new.slot = $slot`; `rotate()` writes one slot-keyed rotation entry. Reactive `auth_invalid` events (which carry an `auth_context_id`) are mapped id‚Üíslot via the same graph-seeded map.
- **`dispatch/executor/liveness.py`**: liveness material lookup goes through the same `SlotResolvingSecretStore`.
- **TestCase schema + `key_hash`**: `attacker_principal: str`, `attacker_slot: str` added (in the key); `auth_context_id` stays (out of the key, `ON MATCH SET`). Planner assembly already projects `principal_label`; it adds `slot` from the AC node.
- **Migration**: `doo engagement migrate-testcase-keys -e <id>` backfills `(attacker_principal, attacker_slot)` from each TestCase's stored `auth_context_id` via the graph and recomputes `key_hash`; on collision (two old TestCases collapse to one new key) it keeps the most-recently-reviewed and marks the other `status=retracted` + `MERGED_INTO`. Pre-production engagements may skip migration and accept one round of duplicate proposals instead.
- **Discovered AuthContexts** carry no `slot` (no declaration). Per #110 the planner already excludes them as attacker candidates (un-armable), so slot-resolution is total over `tc.auth_context_id`. A discovered AC appearing as a *victim* in the confirm loop still has no live material ‚Äî that remains a legitimate `hazard_unresolved` for the `baseline_victim` role, out of scope here.
- **The `tier` overload** (`declared`/`discovered` provenance vs "capability tier") is noted but not renamed here; `slot` is deliberately a third, distinct term.
- **Rotation file on disk** is a host-local secrets cache; format change = delete the file, no graph migration.
