# C3/sink/tenant send-as identity is the Principal; `auth_context_id` is the armable resolution handle

The C3 / sink / C4-tenant assemblers resolve the **send-as identity** as a **Principal**, not as the specific `AuthContext` observed in the evidencing traffic. `_fetch_send_as_auth` (and the tenant analogue) ranks the *Principals* that hit the target endpoint â€” `declared > anonymous > discovered`, tie-break `p.label` â€” picks the top one, and resolves that Principal's armable AC via `_fetch_principal_auth`. The committed `tc.auth_context_id` is therefore the **armable** AC the planner resolved for `(attacker_principal, attacker_slot)` â€” the dispatch-side slot-resolution handle â€” not necessarily the AC that was `OBSERVED_UNDER` the evidencing request. This is the assembler-side mirror of ADR-0049's dispatch-side slot indirection.

## Why

On a real cookie-auth engagement, ADR-0048 reconciliation correctly attached every observed (rotated) session cookie to its declared Principal â€” but the AC nodes stay `tier='discovered'` (different token bytes than the declared `${ENV_VAR}`). 13 discovered ACs carried ~14Ã— the traffic of the 27 declared generations. The C3 target endpoints were hit *only* under those discovered cookies, so after #129 (which correctly projects `slot=None` for discovered ACs) `_fetch_send_as_auth` returned `slot=None` for every gap and `resolve_c3_draft` rejected 24/24 with `attacker_no_slot` â€” even though `(admin, cookie)` / `(monitor, cookie)` / `(system-test, cookie)` were all fully armable.

The send-as identity for a C3 leak-replay is "the actor who reached the input endpoint", and per CONTEXT.md *reached* is defined **as a Principal** (an Endpoint is reached *by a Principal* when an observation under that Principal's AuthContext returned 2xx). The AC is evidence of *who*; which token-snapshot carried the request is incidental. C2/C2b already get this right because they enter via `principal_id` and call `_fetch_principal_auth` (which prefers `tier='declared'`). C3/sink/C4-tenant entered via the observed AC and never went back through the Principal.

## Considered Options

- **Dispatcher-side: widen `build_declared_slot_map` to discovered ACs whose `OF_PRINCIPAL â†’ p.tier='declared'`** â€” rejected. ADR-0049 deliberately scoped `graph_map` to `tier='declared'` so a planner leak (like #129 was) surfaces as `hazard_unresolved` rather than silently arming. Dispatch is the correctness layer; it stays strict.
- **Keep `auth_context_id` = the observed (discovered) AC; add a non-key `observed_auth_context_id` for evidence** â€” rejected as YAGNI. The evidence trail is one Cypher hop from the Principal (`p <-[:OF_PRINCIPAL]- (discovered ac) <-[:OBSERVED_UNDER]- (r) -[:HIT]-> (e)`); nothing reads a denormalised pointer today. Revisit if a consumer needs it.
- **AC-first with post-hoc substitution** (pick the top AC; if discovered âˆ§ `p.tier='declared'`, swap in a declared AC) â€” rejected for the multi-Principal case: when the endpoint was hit by `{discovered AC of declared Principal, discovered AC of discovered Principal}`, an `ac.id` tie-break can land on the un-substitutable row and reject when a substitutable one existed. Ranking by Principal first makes the armable row win deterministically.
- **Slot-agnostic AC resolution** (any armable AC of the Principal, ignoring `token_kind`) â€” rejected: substituting a declared bearer for an observed cookie may not reach the endpoint (different auth mechanism) â†’ false negative. Kept as the last-resort fallback when the Principal declares no credential of the observed kind.
- **Include C4-capability** â€” rejected. A capability `TrustBoundary` is drawn between two **AuthContexts of the same Principal**; the boundary's meaning *is* the claim delta between those specific ACs. Substituting a declared AC would need one at the same capability tier, which is not verifiable here. A capability boundary whose weak side is discovered stays `attacker_no_slot` until the tester declares a credential at that tier.

## Consequences

- **`_fetch_send_as_auth`** is restructured Principal-first: one Cypher ranks the distinct Principals that `HIT` the endpoint (via `OBSERVED_UNDER â†’ ac â†’ OF_PRINCIPAL â†’ p`), ordered `(p.tier='declared') DESC, p.is_anonymous DESC, p.label ASC`; for the top Principal it returns `_fetch_principal_auth(p.id, prefer_token_kind=<the observed ac.token_kind>)`. The sink-pack assembler follows the same path.
- **`_fetch_principal_auth`** gains an optional `prefer_token_kind: str | None` that, when set, adds `(ac.token_kind = $kind) DESC, (ac.slot = $kind) DESC` ahead of the existing `id` tie-break â€” same-kind-default-slot first (the ADR-0049 default-slot convention), same-kind-other-slot next, other-kind last. C2/C2b callers pass `None` â†’ unchanged.
- **`_tenant_auth_contexts`** drops its inline per-tenant AC pick and resolves each tenant's attacker/victim via `_fetch_principal_auth(p.id)` over the tenant's Principals, ranked the same way.
- **`_capability_auth_contexts`** is unchanged (out of scope; see above).
- **`tc.auth_context_id` semantics** (amends ADR-0049): a non-key property holding the *armable* AC the planner resolved for `(attacker_principal, attacker_slot)`. It may point at an AC with no `OBSERVED_UNDER` edge to the target endpoint when the evidencing traffic used a discovered cookie of the same Principal. CONTEXT.md's TestCase-identity entry is updated accordingly.
- **`status='active'` filter stays** on `_fetch_principal_auth`. If a declared Principal's only active ACs are discovered (all declared generations expired, helper mid-rotation), the gap rejects at plan time â€” fail-closed; the race window is one auth-helper transaction.
- **No migration.** Existing TestCases re-resolve on the next `planner propose` (`auth_context_id` is `ON MATCH SET`); `key_hash` is unchanged (it keys on `(attacker_principal, attacker_slot)`, which this does not alter).
